# mdxdb + webapp vs labvault + platform 比較資料

## 1. コンセプトの違い

### mdxdb + webapp: 「人間がデータを整理・閲覧するためのツール」

研究者が手動で実験データを記録し、Web UI で閲覧する。従来型のデータ管理。

```
研究者が手動で記録 → Nextcloud に保存 → Web UI で閲覧
```

- 記録するもしないも研究者次第
- 保存されたデータは人間がブラウザで見に行く
- データは「ファイルとして置いてある」状態

### labvault + platform: 「LLM が実験データを理解・活用するための基盤」

実験の全過程を自動記録し、LLM がデータを横断的に検索・分析・比較できるようにする。**データの主な消費者は人間だけでなく、AI**。

```
実験が自動記録される → LLM が横断検索・分析 → 研究者に知見を返す
```

- Notebook セルの実行コード・変数変化が自動記録される
- データは LLM が理解できる構造で保存される (Embedding, 構造化メタデータ)
- Claude/Gemini に「○○の実験を探して」「条件の違いを比較して」と聞ける

## 2. この設計の違いが生む具体的な差

### 「先月の実験と今日の実験を比較したい」

**mdxdb**: 研究者が Web UI で両方のレコードを開き、手動で条件・結果を見比べる

**labvault**: Claude に聞く
```
「先月の Fe-Cr 成膜と今日の結果を比較して」
→ MCP search で候補を検索
→ MCP compare で条件・結果を自動比較
→ 「温度が200℃違い、膜厚に30nmの差があります」
```

### 「この結果はどのコードで得られた？」

**mdxdb**: 記録されていない。研究者が Notebook を探して遡る

**labvault**: 全セル実行が自動記録済み
```
「AB3F の lattice_a はどう計算された？」
→ CellLog から該当セルのコードと変数変化を自動追跡
```

### 「膜厚と温度の相関を調べたい」

**mdxdb**: 研究者が各レコードのJSONを開いてデータを手動集計

**labvault**: Claude に聞く
```
「temperature_C ごとの thickness_nm の傾向を見せて」
→ MCP aggregate で全レコードから統計を自動集計
```

### 「新しいメンバーが過去の実験を理解したい」

**mdxdb**: 先輩に聞くか、Nextcloud のフォルダを一つずつ開いて探す

**labvault**: Claude に聞く
```
「Fe-Cr 薄膜の実験の概要を教えて」
→ セマンティック検索で関連レコードを収集
→ 条件・結果・メモを横断的にまとめて回答
```

## 3. LLM 最適化のための設計判断

labvault が mdxdb と異なる設計を選んでいる理由は、全て「LLM がデータを活用できるようにする」ためです。

| 設計判断 | mdxdb の方式 | labvault の方式 | なぜ変えたか |
|---------|-------------|----------------|------------|
| **メタデータ保存** | JSON ファイル | Firestore | LLM が高速にクエリ・フィルタするため |
| **検索** | なし | セマンティック検索 (Embedding) | 「結晶性が良い薄膜」のような自然言語で探せるように |
| **セル実行記録** | なし | 自動記録 (CellLog) | 「この結果のコードを見せて」に LLM が答えるため |
| **変数追跡** | なし | namespace diff | LLM が「何が変わったか」を把握するため |
| **MCP サーバー** | なし | 6ツール | Claude/Gemini が直接データを操作するため |
| **Embedding テキスト** | なし | title + tags + conditions + results | LLM が類似実験を見つけるための情報を密に詰めるため |
| **構造化された結果** | 自由形式 JSON | 型付き results/conditions | LLM が比較・集計しやすい構造にするため |
| **レコード間リンク** | なし | link + parent/child | LLM が実験の因果関係を辿るため |

## 4. 長所の比較

### mdxdb + webapp の長所

- **シンプル**: ファイルを Nextcloud に置くだけ。学習コストが低い
- **Web UI**: ブラウザからすぐ見られる。非エンジニアにも使いやすい
- **GCP 不要**: Nextcloud だけで動く。追加コストなし
- **既に運用実績あり**: 現在使えている

### labvault + platform の長所

- **自動記録**: `lab.new()` の 1 行で以降の全操作が自動記録される。記録忘れがない
- **LLM 横断検索**: 「○○の実験を探して」で全データをセマンティック検索
- **LLM 比較分析**: 「条件の違いを比較して」で自動横断比較
- **LLM 統計集計**: 「○○の傾向を見せて」で複数実験を自動集計
- **データ消失防止**: SQLite バッファ + 自動同期で、ネットワーク障害でもデータが消えない
- **再現性**: 全セルの実行コードが記録されるため、結果の導出過程を完全に辿れる
- **チーム横断**: 全メンバーのデータが 1 つの検索可能なプールに統合される
- **低コスト**: GCP 月額約 $1
- **Web UI**: Next.js + shadcn/ui による閲覧・検索・一括アップロード
- **ファイルプレビュー**: VK4 (レーザー顕微鏡) 等の専用デコーダーで画像プレビュー

## 7. 移行状況

- mdxdb の全データ (3ルートレコード + 1,620サブレコード) を labvault に移行済み
- Nextcloud 上の既存ファイルはそのまま利用 (コピーなし、パス参照のみ)
- 条件データ (condition.json) も labvault の構造化 conditions に取り込み済み
- Web UI で閲覧・検索・VK4 プレビューが動作

## 8. まとめ

mdxdb は「研究者がデータを保存・閲覧するツール」として十分に機能しています。

labvault は「**LLM がデータを理解・活用できる基盤**」として設計を刷新したものです。自動記録・セマンティック検索・MCP によるAI連携が加わり、「データを溜める」だけでなく「データから知見を引き出す」ことが可能になります。

既存の mdxdb データは labvault に移行済みで、Nextcloud (ARIM MDX) はそのまま使い続けています。
