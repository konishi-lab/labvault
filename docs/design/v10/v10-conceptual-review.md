# labvault v10 コンセプトレビュー

> 実験物理学者・理論物理学者・PythonプロフェッショナルによるAPI設計/コンセプトレベルの批評。
> 実装のバグではなく「そもそもこのアプローチは正しいのか」に焦点を当てる。

---

## 致命的な指摘（コンセプトの根本に関わる問題）

### C-1. Notebook中心主義の限界 — labvaultのコア価値が現場の10-20%しかカバーしない

**指摘者**: 実験物理学者

labvaultの最大の売りは「Notebookで普通にコードを書くだけで全実行履歴が自動保存」だが、実験物理の作業時間の配分は:

| フェーズ | 時間配分 | labvaultのカバー |
|---------|---------|:---------------:|
| 装置操作・測定 | 50-60% | ほぼなし |
| データ解析 | 20-30% | **ここだけ** |
| 結果整理・議論・論文 | 10-20% | 部分的 |

IPython hooksが活きるのは「Python + Notebookで解析する場合」のみ。多くの学生はまだOriginやExcelで解析しているのが現実。**ファイル管理・メタデータ記録こそがコア価値であり、IPython hooksはオプション機能であるべき。**

---

### C-2. 「3行で始められる」の欺瞞 — GCPインフラ前提との根本矛盾

**指摘者**: 実験物理学者 + Pythonプロフェッショナル

Cookbookの「3行で最小限」は、裏側で以下を前提としている:
- GCPプロジェクトの作成 + Firestoreの有効化
- Nextcloudサーバー（30TB）の用意
- サービスアカウントの作成と認証設定
- `config.toml` の設定

MLflow/W&Bは**SaaS側でインフラを完全に隠蔽**しているから「1行で始められる」が成立する。labvaultの「3行」は実態と乖離したコピーである。

REQUIREMENTS.md の NF05「運用ゼロ」「研究室にDB管理者がいない前提」とも矛盾する。

**提案**: **ローカルオンリーモード（SQLite + ローカルファイルシステムのみ）をデフォルトにし、GCPなしで本当に3行で動くようにすべき。** リモートバックエンドは後から接続する段階的オンボーディング。

---

### C-3. 自動ログと手動入力の二重人格 — 結局どちらが主体なのか

**指摘者**: Pythonプロフェッショナル

「コードを書くだけで全履歴が自動保存」が最大の売りなのに、Cookbookの全パターンで `exp.conditions(...)`, `exp.results[...] = ...`, `exp.status = "success"` の手動操作が必須。

自動ログが記録するのは「セルのソースコードと変数変化」であり、**実験の意味論的な構造（条件・結果）は全て手動入力**。

「自動」と「手動」が異なるレイヤーの情報を扱っているにもかかわらず、ユーザーの認知モデルでは「自動で全部記録されるはずなのに、なぜ手動入力が多いのか」という認知的不協和が生じる。

**提案**: 「LLMのための実験データ基盤」を唯一の設計原則に据え、自動ログ+手動アノテーションの組み合わせを「LLMのための教師データ作成」としてリフレーミングする。

---

### C-4. Record クラスの God Object 化

**指摘者**: Pythonプロフェッショナル

Record クラスが抱える責務:

| 責務 | メソッド群 |
|------|----------|
| データ管理 | add, add_dir, save, get_data, list_data, add_ref |
| メタデータ管理 | conditions, tag, untag, note, status |
| 結果管理 | results (proxy) |
| 構造管理 | sub, children, link |
| 解析管理 | analyses |
| トラッキング | pause_logging, resume_logging, track, track_block, snapshot |
| シリアライズ | to_dict, from_dict |
| ライフサイクル | close, \_\_enter\_\_, \_\_exit\_\_ |

**少なくとも6つの異なる責務が1クラスに集約されている。** 特に「ファイルストレージ操作（add/save）」と「IPython hooks制御（pause_logging/snapshot）」が同一クラスにあることはSRP違反。

**提案**: Facadeパターンで表面APIは維持しつつ、内部を RecordData / RecordStorage / RecordTracker に分離。

---

### C-5. 計算科学のワークフローに根本的に合わない

**指摘者**: 理論物理学者

labvaultのRecordモデルは「1人の実験者が1つの実験を対話的に記録する」前提。計算科学では:

- **DAGワークフロー**: 構造最適化→SCF→バンド計算+フォノン計算（分岐・合流）。ProcessChainは線形チェーンのみ
- **バッチジョブ**: 数千〜数万のジョブをSlurm投入。Record 1つ = 1実験のモデルでは破綻
- **HPC環境**: インターネット接続なし、pip install制限、Notebookなし
- **4文字ID**: 32^4 ≈ 100万通り。高スループットスクリーニングで使い切る可能性あり

**提案**: 計算科学は「labvault自体のモデル拡張」ではなく、**AiiDA/Atomate2のデータをlabvaultにインポートするアダプタ**として設計すべき。labvaultの強みは実験装置→LLM検索パイプラインにある。

---

### C-6. `lab.new()` の暗黙的副作用 — 驚き最小の原則違反

**指摘者**: Pythonプロフェッショナル

`lab.new()` は「レコード作成」という名前だが、実際には:

1. Recordオブジェクト生成
2. SQLiteバッファ書き込み
3. **既存のアクティブなtrackerを暗黙的にdeactivate**（他のRecordのログが止まる）
4. IPython hooks登録、以降の全セル実行をキャプチャ開始
5. SyncManagerがバックグラウンドで同期開始

特に3番が問題。`exp2 = lab.new("実験2")` を実行すると、**ユーザーが明示的に停止指示していない `exp1` のログが暗黙的に停止する。** MLflowでは `mlflow.end_run()` の明示的呼び出しが必要。W&Bでは `wandb.finish()` が必要。labvaultの暗黙的切り替えは便利に見えて、デバッグ困難な問題を生む。

## 重要な指摘

### C-7. GCP + Nextcloud 二重インフラの非現実性

**指摘者**: 実験物理学者

「月額$0.94」はGCPのランニングコストだけ。Nextcloud（30TB）の構築・運用コストが含まれていない。大学の情報基盤センターがNextcloudを提供していなければ、研究室でNASを用意してNextcloudをホストする必要があり、「運用ゼロ」の理念と完全に矛盾する。

---

### C-8. 学生卒業時のデータ引き継ぎ — 最重要課題への対応不足

**指摘者**: 実験物理学者

実験物理の研究室で**最も深刻な問題**は学生卒業時のデータ散逸。labvaultに不足しているもの:

- **暗黙知の記録**: 装置の「コツ」（「Ar流量安定前に電力を上げるとアーキング」等）
- **サンプルの物理的所在**: 「2号館3階デシケーター内」
- **背景・校正データの紐付け**: バックグラウンド測定、Si標準、装置校正記録

---

### C-9. メソッドチェーンと即時永続化の矛盾

**指摘者**: Pythonプロフェッショナル

`exp.tag("XRD").conditions(T=500).note("memo")` — メソッドチェーンは「最後にまとめて確定」を示唆する（Builderパターン）。しかし実際には各呼び出しでSQLiteに即時書き込みされる。見た目と挙動のギャップがある。

**提案**: (A) 即時永続化を維持するならチェーンを推奨パターンから外す、(B) チェーンを残すなら明示的な `exp.commit()` で一括永続化する。

---

### C-10. conditions/resultsのスキーマ不在問題

**指摘者**: 理論物理学者 + Pythonプロフェッショナル

`conditions` が `dict[str, Any]` のフリーフォームであることの問題:

- チーム内の表記揺れ（`temperature_C` vs `temp` vs `T`）
- 型不一致（`int` vs `str`）がFirestoreクエリを壊す
- `conditions()` は何度でも上書き可能。途中経過が失われる
- テンプレートの使用は任意。チーム全体のスキーマガバナンスが不在

MLflowでは `mlflow.log_params()` でイミュータブルに記録する。Sacredでは `@ex.config` で宣言的に定義する。

---

### C-11. 差別化ポイントの脆弱性

**指摘者**: Pythonプロフェッショナル

labvaultの差別化は:
1. IPython hooks自動ログ — しかし意味論的情報は手動
2. LLM検索（MCP経由）— GCPインフラ依存
3. 材料科学特化パーサー — ニッチ

問題は**統合されたストーリーとして一貫していない**こと。MLflowは「MLライフサイクル管理」、W&Bは「ML実験コラボレーション」、DVCは「データバージョン管理」。labvaultは「実験ログ」なのか「データ管理」なのか「LLMアシスタント」なのか、主軸が不明確。

---

### C-12. 既存ELN/計算科学ツールとの差別化不明確

**指摘者**: 実験物理学者 + 理論物理学者

- **LabArchives/eLabFTW**: 非プログラマでも使える。Web UIあり。labvaultはPython必須でWeb UIなし
- **AiiDA**: DAGワークフロー管理、計算プラグイン、ジョブスケジューラ統合。計算科学では圧倒的
- **Materials Project/NOMAD**: FAIRデータ共有。labvaultは「スコープ外」

labvaultの真の差別化は「IPython hooks + LLM検索」だが、Python解析をする研究室にしか刺さらない。

---

### C-13. テンプレートのフィールド不足

**指摘者**: 実験物理学者

- **XRD**: スリット条件（発散/受光/散乱）、入射角（GI-XRD）が欠落
- **SEM**: プローブ電流、スティグマ条件、真空度が欠落。EDS定量分析に重要
- **SQUID**: 最近はSI単位（T）も使う。磁場掃引速度が欠落
- **未対応測定**: XPS(VAMAS), AFM(.spm), XRF, DSC/TG, FT-IR(JCAMP-DX), EPMA が全くない

---

### C-14. Notebookでのセル再実行の非冪等性

**指摘者**: Pythonプロフェッショナル

`exp.tag("XRD")` は冪等（2回実行しても同じ結果）だが、`exp.note("メモ")` は非冪等（2回実行するとメモが重複）。セル再実行が日常的なNotebookで、どの操作が冪等でどの操作が非冪等かをユーザーが意識する必要がある。

---

### C-15. バックグラウンド同期の「沈黙の失敗」

**指摘者**: Pythonプロフェッショナル

同期失敗が `warnings.warn()` で通知される設計は、Notebook環境では**ほぼ確実にユーザーに見逃される**。W&Bは `wandb: WARNING Network is unavailable` を明示的にログに出す。オフライン遷移時に一度だけ明確な通知を出すべき。

## 改善提案

### C-16. PIと学生の温度差 — 段階的導入戦略の必要性

**指摘者**: 実験物理学者

全機能を最初から使わせるのではなく段階を踏むべき:

| Phase | 対象 | 機能 | ハードル |
|-------|------|------|---------|
| Phase 0 | 全員 | ファイル保存だけ（`labvault add ID file`） | USBメモリより楽かどうか |
| Phase 1 | 全員 | テンプレート・タグ・条件入力 | 5分の追加作業 |
| Phase 2 | Python使う人 | Notebook統合（IPython hooks） | Pythonスキル |
| Phase 3 | 全員 | LLM連携 | データ蓄積後 |

**Phase 0の「ファイルを特定の場所に入れるだけ」すらUSBメモリ運用より楽でないと、誰も使わない。**

---

### C-17. パーサーより装置ログ自動取り込みを優先

**指摘者**: 実験物理学者

.rasファイルのヘッダーパースより、**装置が自動出力するプロセスログ**（電力・温度・圧力の時系列データ）をまるごと取り込む方が再現性にとって重要。手入力では絶対に記録しきれない情報が含まれている。

---

### C-18. 計算科学はAiiDA連携で解決すべき

**指摘者**: 理論物理学者

`type="computation"` をサポートし `add_ref(location="TSUBAME:...")` の例を示す以上、計算科学者が使おうとするのは自然。しかし:
- DAGワークフロー表現力なし
- バッチジョブ管理なし
- 計算パラメータの構造化不可
- 計算環境の自動記録なし

AiiDA/Atomate2のデータをlabvaultにインポートするアダプタとして設計し、**実験データと計算データの横断検索・LLM解析に特化**する方が効率的。

## コンセプトの再定義案

3つの専門家の批評を統合すると、labvaultに必要なのは以下の**コンセプト再定義**:

### Before（現在のコンセプト）
> Notebookで普通にコードを書くだけで、全実行履歴がLLMに理解可能な形で自動保存される

### After（再定義案）
> 実験データを「入れるだけ」で消えない・見つかる・引き継げる。Pythonを使うなら解析履歴も自動で残る。溜まったデータはLLMが横断的に検索・解析する

**変更のポイント**:
1. **「Notebook」から「データ」への主語転換** — Notebookを使わない人も対象
2. **「自動保存」から「入れるだけ」への動詞転換** — 最小限の手間を強調
3. **「消えない・見つかる・引き継げる」の追加** — 研究室の本当のペイン
4. **IPython hooksはボーナス扱い** — 「Pythonを使うなら」という条件付き
5. **LLMは蓄積後の価値** — 最初の動機にはしない

### これに基づく設計変更の方向性

| 変更 | 内容 |
|------|------|
| デフォルトモード | **ローカルオンリー**（SQLite + ローカルFS）。GCPは後から接続 |
| コア価値 | ファイル管理 + メタデータ記録。IPython hooksはオプション |
| Record分割 | Facade維持。内部を Data / Storage / Tracking に分離 |
| 計算科学 | labvault本体ではなくAiiDAアダプタで対応 |
| 段階的導入 | Phase 0（ファイルのみ）→ Phase 1（メタデータ）→ Phase 2（Notebook）→ Phase 3（LLM） |
| メソッドチェーン | 明示的 `exp.commit()` で一括永続化。または即時永続化ならチェーン非推奨 |
| `lab.new()` | IPython hooksの起動は `auto_log=True`（デフォルト）で明示。前のtrackerの暗黙停止を廃止 |
