## 2. 新たに見える問題（Python/Notebook日常 + Nextcloud大学提供）

### N-1. 装置制御スクリプト(.py)とIPython hooksの断絶 【高】

装置制御がPythonスクリプト（`.py`）の場合、IPython hooksは効かない。`@exp.track` は関数単位であり、長時間の装置制御（手続き的で関数に分解しにくい）には不十分。

**提案**: 装置制御スクリプト向けの時系列ロガーAPIを追加:
```python
exp.log_value("temperature_C", 301.2)         # タイムスタンプ付き値記録
exp.log_event("deposition_start", "RF ON")     # イベント記録
```

### N-2. 長時間測定のセッション管理 【高】

一晩放置のSQUID測定をNotebookから実行 → カーネルがタイムアウト/OOMで死ぬ → Recordがrunningのまま放置 → セルログが中断。

**対応**: `lab.get("AB3F", auto_log=True)` で既存Recordに再接続し、IPython hooksを再開する。新APIは不要。

### N-3. セル再実行の冪等性保証 【高】

Notebookが日常 = セル再実行が高頻度。以下を冪等にすべき:
- `exp.note("text")` — 同一テキストの重複防止（同セルのexecution_count管理）
- `exp.add("file.csv")` — 同一ファイルの重複防止（SHA256比較）
- セルログ — execution_countをキーに上書き更新

### N-4. 測定Notebookと解析Notebookの分離 【高】

よくあるパターン:
1. `measurement.ipynb` で装置制御・データ取得
2. `analysis.ipynb` で解析・可視化（同じRecordに追記）

**対応**: `lab.get("AB3F", auto_log=True)` で既存Recordを取得し、セルログを追記する。セッション分離はNotebook名やカーネルIDで内部的に自動判別する。新APIは不要。
```
sessions: [
  {notebook: "measurement.ipynb", cells: [...]},
  {notebook: "analysis.ipynb", cells: [...]},  ← lab.get(auto_log=True) で自動分離
]
```

### N-5. 装置制御と解析の並行アクセス 【中】

装置制御Pythonプロセスと解析Notebookが同時に同一Recordに書き込む場合、SQLiteロック競合やFirestore上書き競合が発生。

**提案**: 「装置制御Recordと解析Recordは分けて`link()`で紐付け」を推奨パターンとするか、Recordへの書き込みをセッション単位で分離。

### N-6. Nextcloud大学提供ゆえの制約 【中】

- LDAP連携で個人アカウント管理 → 卒業時にアカウント無効化
- 共有フォルダの権限ポリシーが大学に依存
- WebDAV APIにアプリパスワードが必要な場合がある

**提案**: 研究室共有フォルダ方式（グループフォルダ）を推奨パターンとして明記。卒業時のデータ所有権移転手順を文書化。

## 3. コンセプト再定義（最終版）

### v10現在
> Notebookで普通にコードを書くだけで、全実行履歴がLLMに理解可能な形で自動保存される

### 初回レビュー後の再定義案
> 実験データを「入れるだけ」で消えない・見つかる・引き継げる。Pythonを使うなら解析履歴も自動で残る。溜まったデータはLLMが横断的に検索・解析する

### 最終修正案

> **Python/Notebookで実験する研究室のための実験データ基盤。**
> **測定から解析までのコード・データ・条件が自動で記録され、蓄積されたデータをLLMが横断検索・解析する。**

**ポイント**:
- 対象を「Python/Notebookで実験する研究室」に明確に絞る
- IPython hooksをコア価値として復帰（「自動で記録」）
- ただし「全実行履歴」という過大表現は避け「コード・データ・条件」と具体化
- 手動入力（conditions）の存在も暗示（「条件が記録され」）
- LLMは蓄積後の価値として後置
